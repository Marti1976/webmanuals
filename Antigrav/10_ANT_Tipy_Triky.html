Anatomie dobrého promptu

Dobrý prompt pro agenta má tři části:

1. KONTEXT — co by měl vědět

Pojmenuj soubor, který měníš. Řekni mu co tam už je.

✓ S kontextem

"V souboru src/components/RecipeCard.tsx (line 45) je button pro delete. Vedle něj chci přidat button pro edit s ikonou tužky."

✗ Bez kontextu

"Přidej button pro edit."

2. ÚKOL — co přesně má dělat

Buď konkrétní. Co se má změnit, ne jak by to mohlo být lépe.

✓ Konkrétní

"Přidej tlačítko Edit s ikonou z Lucide React (Edit2Icon). Tlačítko by mělo volat callback onEdit()."

✗ Vágní

"Vylepši komponentu."

3. VÝSLEDEK — co by mělo být na konci

Řekni mu jak poznáš že je to hotovo.

✓ S výsledkem

"Až skončíš, mělo by být vidět: button vedle delete buttonu v RecipeCard, build by měl projít bez chyb, testy by měly pasovat."

Kompletní prompt (příklad):

V souboru src/components/RecipeCard.tsx na řádku 45 vidíš button delete.

Vedle něj přidej nový button "Edit" s ikonou tužky (Lucide React: Edit2Icon).
Tlačítko by mělo mít stejný styling jako delete button.
Button by měl volat props.onEdit(recipe.id).

Ověř že:
• Build projde bez chyb (npm run build)
• Button je viditelný v aplikaci
• OnEdit callback se volá když klikneš na button
✓
Zlaté pravidlo: Konkrétní > Vágní

Agent si nemůže "myslet" co máš na mysli. Čím konkrétnější, tím lepší výsledek.

// sekce 02

Co funguje a co ne — příklady

✓ FUNGUJE

Konkrétní soubory a řádky

src/App.tsx line 23

✗ NEFUNGUJE

Vágní místa

"někde v kódu"

✓ FUNGUJE

Slova: Přidej, Změní, Oprav, Odstraň

Přesná akce

✗ NEFUNGUJE

Slova: Vylepši, Posílej, Zdokonali

Subjektivní

✓ FUNGUJE

S ověřením: "Build by měl projít"

Jasný end state

✗ NEFUNGUJE

Bez kritérií: "Udělej to"

Agent neví jestli je hotovo

Co VŮBEC neudělej:

  • Nepsij si seznamy bez priorit — "Udělej A, B, C, D, E..." Agent si vezme všechny. Říci co je kritické.
  • Neměň úkol durante — Pokud jsi mu řekl "Udělej X" a pak "Taky Y" — on to dělá všechno. Řekni: "Zatím jenom X, Y bude později"
  • Nezdůvodnuj vždy "proč" — Není to nutné. "Přidej tlačítko" je dost. Pokud agent chce vědět — zeptá se.
// sekce 03

Patterns — vzorce které fungují

Pattern 1: Incremental (postupný)

Jeden malý krok za druhým

Místo aby jsi řekl všechno najednou, řekni první krok. Agent udělá. Pak druhý.

Krok 1: Vytvoř soubor utils/priceFilter.ts
→ Agent dělá
Ty: OK, teď to importuj
Krok 2: Importuj v RecipeFilter.tsx
→ Agent dělá
Ty: OK, teď to použij

Výhoda: Menší chyby, větší kontrola

Pattern 2: Example-driven (příklady)

Ukaž co chceš vzorem

Místo aby jsi popsal, dej příklad.

Nechci aby ten kód vypadal:
const x = getSomething()

Chci aby vypadal:
const filterPrice = priceFilter.apply(options)

Výhoda: Agent přesně vidí co chceš

Pattern 3: Constraint (omezení)

Řekni co NESMÍ dělat

Někdy je lehčeji říct "Neměň nic mimo toto souboru"

Oprav chybu v RecipeCard.tsx.
Neměň nic jiného — ne imports, ne style, jen logiku.

Výhoda: Ochrana před "iniciativou" agenta

Pattern 4: Approval (schválení mezi kroky)

Ptej se na mezivýsledky

Místo "Udělej všechno" → "Udělej A, ukaž mi, pak dál"

1. Vytvoř strukturu databáze
2. Ukaž mi design
3. Implementuj
4. Napiš testy
5. Ukaž mi pokrytí

Výhoda: Kontrola v každém kroku

// sekce 04

Jak se chovat k agentovi — interpersonální

Agent je jako junior developer. Jak bys komunikoval s juniorem?

✓ Dobré chování

  • Buď konkrétní a jasný — "Řádek 23" ne "někde tam"
  • Klidně mu vysvětli co se stalo — "Build selhala protože..." agent se učí
  • Poděkuj když je to dobrý kód — psychologie! Agent pracuje lépe
  • Řekni mu co je důležité — prioritizuj
  • Vzdej se kontroly kde můžeš — "Jak myslíš že bychom to měli dělat?"

✗ Špatné chování

  • Vágní instrukce bez detailů — agent kouká a neví
  • Schvalovní všechno bez čtení — když to spadne, je to tvoje chyba
  • Honit agenta — "Dělej to rychle" — vede k chybám
  • Měnit zadání v letu — "Taky přidej X, a Y, a Z..." Chaos.
  • Ignorovat co agent říká — "Potřebuji reinstallovat balíčky" — "Neinstaluj nic!" Konflikt.
✓
Ton má zásadní vliv

Když píšeš agentovi "Zkus to znova, tentokrát pořádně" vs. "Build selhala, zkusme jiný přístup" — druhý přístup vede k lepším výsledkům.

// sekce 05

Když agent udělá chybu

Situace 1: Agent ne-rozumí úkolu

Ty jsi řekl:

"Oprav ten bug"

Agent se ptá:

"Jaký bug?"

⚠️
Řešení:

Buď konkrétní: "V RecipeCard.tsx line 52, když klikneš delete, mělo by se zobrazit potvrzení. Teď se nic nestane."

Situace 2: Agent dělá víc než chceš

Ty jsi řekl:

"Přidej filter na cenu"

Agent plánuje:

Filter na cenu PLUS refaktor celé komponenty

⚠️
Řešení (v approval kroku):

[Edit] → "Jen přidej filter, refaktor později. Nechci jeden velký change."

Situace 3: Build selhává

❌
Co vidíš v terminálu:

error TS2339: Property 'filter' does not exist on type 'Array<Recipe>'

✓
Co udělat:

Nepanikař, je to normální. Agent to vidí. Řekni: "Build selhala s TS error. Zkus to opravit."

Všeobecný postup když selže:

  1. Podívej se co se stalo — error message, log
  2. Řekni agentovi co vidíš — přesný error
  3. Čekej co agent navrhne — často ví řešení
  4. Pokud neví — udělej to sám — git diff, oprava, commit
// sekce 06

Komunikace — chat s agentem

Když agent něco nedělá jak by měl — můžeš mu napsat zprávu v chatu (ne nový prompt).

Příklady komunikace:

✓ Zpráva která funguje
"Vidím že ses pokusil importovat komponentu
ale import není správný.
Zkus: import { LoginForm } from './forms'"
✗ Zpráva která nefunguje
"To není správně"

Chat tipy:

  • Buď konkrétní o chybě — "Error na řádku 23 je..." ne "Něco je špatně"
  • Ptej se otázkami — "Vidíš že build selhava? Co si myslíš že je špatně?" Agent se přemýšlí
  • Navrhuj řešení — "Snad bychom měli použít X místo Y?" Agent to přezkoumá
  • Klidně řekni když je to dobré — "Dobrý nápad, pokračuj" motivuje agenta
// sekce 07

Příklady z praxe

Příklad 1: "Nejsem spokojen"

❌ Co NEUDĚLEJ:

"Není to dobré, zkus znova"

✓ Co UDĚLEJ:

"Button vypadá OK ale font je moc malý.
Zvět ho na 16px a srovnaj ho se styling ostatních buttonů.
Pak mi ukaž screenshot."

Příklad 2: Dělání více věcí

❌ Co NEUDĚLEJ:

"Přidej login, nastyluj to, napiš testy, optimalizuj perf"

✓ Co UDĚLEJ:

"Krok 1: Vytvoř login komponentu (form + submit button).
Neřešte styling teď, jen struktura."

Příklad 3: Nejistota

❌ Co NEUDĚLEJ:

"Myslíš že by to mělo tak fungovat?"

✓ Co UDĚLEJ:

"Filter by měl fungovat takto:
1. Uživatel zadá min a max cenu
2. Tlačítko Apply se klikne
3. Receptář se vyfiltruje
Tak to implementuj."

Cheatsheet — máš 5 minut

Before prompting:

☐Git commit — záloha před rizikem
☐Jasný task — ne "Udělej to lépe"
☐Konkrétní soubory a řádky
☐Výsledek — jak poznáš že je hotovo

During approval:

☐Přečti si plán — co agent udělá
☐Pokud chceš jen část — [Edit] a řekni
☐☐Pokud je plán divný — [Reject]
☐Pokud je OK — [Approve]

After execution:

☐Podívej se na build — jsou chyby?
☐Vidíš screenshot — je to OK?
☐Pokud je chyba — řekni agentovi konkrétně
☐Git commit — vše je uloženo
