// sekce 01

Co je Editor View a co není

Antigravity má dva různé módy práce. Právě teď se zaměříme na Editor View — to je ten tradičnější, více "pod tvou kontrolou".

Editor View = VS Code + bočný agent panel

Levá strana: klasické VS Code — soubory, struktura projektu, editace kódu, všechno co znáš. Pravá strana (nebo v panelu): agent kterého můžeš psát. Můžeš sám kódovat, agent ti pomáhá — nebo mu dáš úkol a on kóduje zatímco ty čekáš.

✅ Co Editor View JE
✓Známé VS Code rozhraní
✓Můžeš kódovat sám, agent v panelu
✓Agent vypomáhá, neovládá
✓Máš vždy poslední slovo
✓Dobré pro kódování s pomocí AI
❌ Co Editor View NENÍ
✗Není to Agent-first (to je Manager View)
✗Nejde tu paralelně více agentů
✗Agent neví co chceš dok ho explicitně neřekneš
✗Není to AI Studio chatbox
💡
Proč začínat s Editor View?

Pokud jsi měl negativní zkušenost s AI Studiem kde agent dělal věci bez tvého vědomí — Editor View je pro tebe bezpečnější volba. Máš přehled, máš kontrolu. Později můžeš zkusit Manager View pokud budeš chtít více autonomie.

// sekce 02

Rozhraní — kde co je

Antigravity Editor View je upravená verze VS Code. Pokud znáš VS Code, půjde ti to snadno. Pokud ne, tady je orientace:

Levá strana — editor a files (klasické)

  • File Explorer — stromová struktura projektu. Všechny složky a soubory tvého receptáře.
  • Search — hledání v souborech (Ctrl+Shift+F / Cmd+Shift+F)
  • Source Control — Git integrace. Uvidíš změny, commity, větve.
  • Extensions — VS Code rozšíření — stejná jako klasické VS Code.

Uprostřed — editory a terminal

  • Tab souboru — otevři soubor a edituj. Agent v panelu vidí co edituješ.
  • Terminal — spusť příkazy (npm run dev, build, git atd.). Agentovi můžeš říct "spusť build" a on to zadá terminálu.
  • Preview — pokud jsi otevřel HTML/Markdown, vidíš náhled.

Pravá strana — agent panel (nové v Antigravity)

  • Chat s agentem — napíšeš task, agent mu rozumí v kontextu celého projektu, může se ptát dál.
  • Artifacts panel — agent zde zobrazuje to co dělá: plány, diffs kódu, screenshoty (pokud má Chrome extension).
  • Status agenta — víš jestli agent čeká, pracuje, nebo čeka na tvé schválení.
┌────────────────────────────────────────────────────────────┐ │ Antigravity IDE — Editor View │ ├────────────────────────────────────────────────────────────┤ │ File Explorer │ src/App.tsx │ Agent Chat │ │ ├─ src/ │ ........................│ "Přidej button" │ │ ├─ public/ │ return <div>... │ ........................│ │ │ ├─ manifest │ const [count] ... │ Artifacts: │ │ └─ package.json│ │ - Plan (diff) │ │ │ .................... │ - Waiting for OK │ │ SEARCH │ Terminal │ [Approve] [Edit] │ │ │ $ npm run dev... │ │ └────────────────────────────────────────────────────────────┘

Klávesové zkratky (stejné jako VS Code)

// sekce 03

Jak agent v Editor View funguje

Přihlášeníš si v Antigravity, otevřeš Editor View, a vidíš agent panel. Co se stane když mu cokoliv řekneš?

Typický workflow agenta

1
Ty napíšeš task (prompt)

Příklad: "Přidej v komponenty RecipeCard.tsx tlačítko pro oblíbené recepty (heart ikona)"

Agent si přečte projekt, vyhledá relevantní soubory, rozumí kontextu.

2
Agent PLÁNUJE (Plan Artifact)

Zobrazí se plán co bude dělat:

📋 Plan:
- Otevřu RecipeCard.tsx
- Přidám import Lucide Heart ikony
- Přidám state pro isFavorite
- Rendernu button s ikonou na řádku 45
- Uložím soubor

Klíčové: Agent se PTÁŠ na schválení. Nečeká jen tak.

3
Ty schválíš nebo odmítneš

V Artifacts panelu jsou tlačítka [Approve] [Edit] [Reject].

Approve: Agent pokračuje.
Edit: Napíšeš feedback, agent se přizpůsobí.
Reject: Agent se zeptá co jinak.

4
Agent PROVÁDÍ (Execution)

Změní soubory, spustí příkazy, vše se udělá. V terminálu vidíš output.

✓ RecipeCard.tsx změněn
✓ Dependency instalován (react-icons)
✓ Build úspěšný

5
Agent OVĚŘUJE (Verification)

Pokud má Chrome extension nainstalovanou, spustí aplikaci v prohlížeči, udělá screenshot, ukaže že tlačítko se zobrazuje správně.

Nebo jen sám řekne: "Hotovo, 3 soubory změněno, žádných chyb"

⚠️
Klíčový rozdíl od AI Studio

Agent si nemůže jen tak "rozmyslet" že pokračuje sám. V každém kroku se ptá. To je tvá ochrana — pokud agent chce dělat věci mimo task, můžeš ho zastavit.

// sekce 04

Artifacts — co to je a proč je to důležité

Artifacts jsou věci které agent vytváří a mezi vámi sdílí. Nejde o "raw tool calls" nebo technické hlášky — jde o věci které chypeš i ty.

Jaké Artifacts agent generuje?

  • Plan (Plánování) — co bude agent dělat, krok za krokem. Vidíš to předtím než se stane.
  • Code Diff — změny v souborech. Barevně zvýrazneno co se změnilo (+zelená, -červená).
  • Screenshot — pokud agent spustil aplikaci v prohlížeči, vidíš jak to vypadá.
  • Task List — seznam úkolů pro agent nebo progress report.
  • Error Report — pokud něco selhalo, agent to dokumentuje s vysvětlením.

Proč na Artifacts záleží?

V AI Studiu jsi viděl jen výsledky — a případně jsi si všimla až když už bylo pozdě. Artifacts ti dávají vizibilitu a kontrolu. Vidíš přesně co se změní dřív než se to změní.

✓
Jak Artifacts fungují v praxi

Agent zobrazí Plan → ty si ho přečteš → buď klikneš Approve (pokračuj), nebo comentuješ jako v Google Docs ("ne, nechci smazat DB, jenom přidej nový field"). Agent vidí tvůj komentář a přizpůsobí se.

// sekce 05

Approval workflow — jak se agent ptá

Toto je ta část která tě ochrání před problémem z AI Studio. Agent se MUSÍ ptát než něco významného udělá.

Režimy schválení (Approval Modes)

V nastavení Antigravity si můžeš zvolit jak přísný máš být s agentem:

🟢 Agent-Assisted (doporučeno)

Agent sám rozhoduje co je bezpečné. Drobné změny — bez ptaní. Větší věci — čeká na schválení.

🟡 Review-Driven

Agent plánuje, ty schválíš plán, pak provádí. Nejbezpečnější.

🟡 Hands-Off

Agent pracuje víc autonomně, ale hlásí ti co udělal. Pro zkušené.

🔴 Full Autonomy

Agent si dělá co chce. Riziko. Nepoužívej pokud máš trauma z AI Studio.

Příklad Approval Workflow

Ty: "Refaktoruj soubor utils.ts abys oddělil logiku" Agent (v panelu): ┌──────────────────────────────────────────┐ │ PLAN ARTIFACT │ ├──────────────────────────────────────────┤ │ Budu: │ │ 1. Extrahovat sortRecipes() do helper │ │ 2. Vytvořit nový soubor helpers.ts │ │ 3. Aktualizovat importy v App.tsx │ │ 4. Spustit testy (npm run test) │ │ │ │ [✓ Approve] [✎ Edit] [✗ Reject] │ └──────────────────────────────────────────┘ Ty: [klikneš ✓ Approve] Agent: (pracuje) [✓] helpers.ts vytvořen [✓] utils.ts refaktorován [✓] Importy aktualizovány [⏳] Spouštím testy... Chyba: Jeden test selhal. Opravuji. [✓] Vše hotovo. Artifacts: - Diff (ukaž změny) - Screenshot (výsledek)
⚠️
Nikdy neklikej "OK na všechno"

Approval je jednocestný. Pokud schválíš špatný plán, agent to udělá. Agentu se pak budou opravovat věci manuálně. Čti vždy co agent plánuje, i když to trvá 30 sekund.

Vs. AI Studio — co se změnilo

Pojďme si srovnat tvůj problém z AI Studio s tím jak funguje Antigravity Editor View.

Situace Google AI Studio Antigravity
Agent vidí tvůj projekt? Jen co zkopíruješ ✓ Vidí vše, celý projekt
Agent si "rozmyslí" další kroky? ✓ Ano, dělá věci bez ptaní ✗ Ne, vždy se ptá
Vidíš PŘED tím co agent udělá? Jen částečně ✓ Plan Artifact, vše vidíš
Můžeš zastavit agent uprostřed? Těžko, už to dělá ✓ Klidně, čeká na schválení
Jak se vrátíš z chyby? Ručně, 3 týdny práce pryč ✓ Git undo, zálohy
Cena? Zdarma Zdarma
💡
Konkrétně k tvé situaci — smazání databáze

V Antigravity by to vypadalo takto: Agent v planu řekne "Smaž databázi a nahraď ji novou strukturou". Ty to vidíš, a klikneš [Reject] nebo [Edit]. Databáze se NEZMĚNÍ pokud ty to neschválíš. Bod.

// sekce 07

Psaní promptu agentovi

Jak psát task aby agent dělal přesně to co chceš? Stejná pravidla jako u Gemini CLI, ale s jedním rozdílem — Antigravity agent vidí tvůj kód.

Vzorce které fungují

✓
Specifický task s kontextem

"V komponenty RecipeCard.tsx, vedle tlačítka Delete (řádek 52), přidej nové tlačítko Edit s ikonou tužky. Tlačítko by mělo volat function onEdit() kterou dostane component skrz props."

✓
S výsledkem na konci

"Refaktoruj src/utils/api.ts: oddělíme fetch logiku do samostatného souboru api-client.ts. Výsledek: utils/api.ts bude mít jen business logiku, api-client.ts bude mít HTTP klienta. Testy by měly projít."

⚠️
Vyhni se vágním tasků

❌ "Vylepši kód"
❌ "Oprav to"
❌ "Udělej to lépe"

Agent neví co znamená "lépe". Buď konkrétní.

Komunikace s agentem během práce

Pokud se agent ptá dál (chatuj s ním), odpovídej konkrétně:

  • "Ano, přidej i error handling" — konkrétní
  • "Ne, nechci smazat ten soubor, jenom ho refaktoruj" — jasné hranice
  • "Zobraz mi diff předtím" — máš kontrolu
// sekce 08

Praktický příklad — krok za krokem

Představme si že otevřeš Antigravity, načteš svůj projekt receptáře a chceš aby agent přidal jednu novou feature.

Příklad: "Přidej filtr na cenu receptu"

A
Ty napíšeš do Agent panelu:

"V komponenty RecipeFilter.tsx přidej nový slider input pro filtrování ceny receptu. Minimálně 0 Kč, maximálně 500 Kč. State ulož do existujícího state objektu filterOptions. Nezapomeň na onChange handler."

B
Agent Creates PLAN Artifact

v panelu se zobrazí:

📋 PLAN - Přidání price filter
1. Otevřu soubor RecipeFilter.tsx
2. Najdu state objektu filterOptions
3. Přidám priceMin a priceMax do stavu
4. Rendernu Slider komponentu (2x)
5. Přidám onChange handlery
6. Uložím soubor

Files: 1 změněn
Build: OK (bez dependencí)

Status: ⏳ Čeká na tvé schválení

C
Ty schválíš plán

V Artifacts panelu: [✓ Approve] — klikneš.

D
Agent PROVÁDÍ (Execution)

Terminal se aktivuje, vidíš co agent dělá:

$ nano src/components/RecipeFilter.tsx
✓ Soubor otevřen a aktualizován
✓ State objekty přidány
✓ Slider komponenty vyrenderovány
✓ Handlers propojeny
$ npm run build
[webpack] Build OK
E
Agent OVĚŘÍ (Verification)

Agent (pokud má Chrome extension) spustí aplikaci a udělá screenshot:

🌐 Spouštím aplikaci v prohlížeči...

✓ Aplikace běží na http://localhost:3000

✓ Price Slider viditelný v UI

✓ Interakce OK — slider se pohybuje 0-500

F
Agent hlásí hotovo

Artifacts panel zobrazí:

  • Diff: Právě co se změnilo v RecipeFilter.tsx
  • Screenshot: Jak vypadá aplikace s novým filtrem
  • Summary: "Hotovo, 1 soubor změněn, 0 chyb, build OK"
G
Ty schválíš nebo edituješ

[✓ Done] — task se uzavřel, agent čeká na nový.
nebo
[✎ Edit] — napíšeš feedback a agent to opravuje.

💡
Pozorování z příkladu

Vidíš PLÁN předtím než se cokoliv změní. Vidíš DIFFU co se změnilo. Vidíš SCREENSHOT výsledku. Má jsi šanci zastavit v kterémkoliv kroku. To je kontrola kterou jsi měla z AI Studio.
