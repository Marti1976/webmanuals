Co agent vidí v tvém projektu

Antigravity agent pracuje v Workspace ve kterém ho otevřeš. Vidí všechny soubory ve všech přidaných složkách.

Co vidí agent?

  • Všechny soubory v /src, /public, package.json, vše
  • Obsah tvého terminálu a výstupy příkazů
  • Git historii (commity, větve)
  • Obsah .env souboru (pokud není ignorován!)
  • Obsah jakéhokoliv souboru co bude otevřený v editoru
🔴
Nikdy neukládej hesla či klíče v .env nebo v kódu

Pokud je tam API klíč, heslo k databázi nebo cokoliv citlivého — agent to vidí a může to poslat Google serverům. Vždy używaj .env který je v .gitignore a v Antigravity project settings ho ignorovat.

Co agent NEVIDÍ?

  • Soubory mimo tvůj Workspace (v jiných složkách)
  • Ostatní projekty které nejsou v Workspace
  • Tvůj obsah v prohlížeči (jen pokud máš Chrome extension a ty mu řekneš aby se tam podíval)
  • Obsah tvého disku mimo projekt
// sekce 02

Approval workflow — jak funguje schválení

Toto je tvoje hlavní ochrana. Agent si nemůže jen tak "rozmyslet" co udělá — musí tě požádat.

Kroky schvalovacího procesu

1️⃣ Agent vidí task

Ty napíšeš: "Přidej filter na cenu". Agent si to přečte.

2️⃣ Agent tvoří PLÁN

Zobrazí se v Artifacts: "Změní soubor X na řádku Y, přidá komponenty Z". Stále nic není změněno.

3️⃣ TY schválíš nebo odmítneš

Vidíš plán a rozhodneš se. Tlačítka: [Approve] [Edit] [Reject]

4️⃣ Agent PROVÁDÍ (jen pokud yes)

Změny se aplikují, vidíš to v terminálu, vidíš diffs.

5️⃣ Agent ověřuje (Verification)

Spustí build, screenshoty, ukazuje že to funguje.

V Review-Driven režimu (doporučeno)

Agent si nemůže nic vzít jako samozřejmost. Každý krok musí být schválen. Není "iniciativy" která by tě překvapila.

✓
To je rozdíl od AI Studio

V AI Studiu agent viděl tvůj kód, řekl si "hmm, k tomu by se hodilo i tohle" a dělal to bez ptaní. V Antigravity: "Chceš aby já taky udělal X?" A ty můžeš říct "NE, jenom Y!"

// sekce 03

Allow/Deny listy — co smí agent dělat

Kromě schválení jednotlivých akcí můžeš agentovi permanentně určit hranice.

Kde to nastavit?

Settings → Agent → Permissions → Allow/Deny Lists

Allow List — co agent SMÍ dělat

Pokud je Allow list zapnutý, agent může dělat POUZE co tam máš:

Allow: - modify files in /src - modify files in /public - run: npm run build - run: npm run dev - Deny: delete files - Deny: modify package.json - Deny: run npm install

Efekt: Agent může měnit kód, ale NEMŮŽE instalovat balíčky ani mazat soubory. Ochrana!

Deny List — co agent NESMÍ dělat

Opačné — co konkrétně je zakázáno:

Deny: - delete any files - modify database.json - modify .env - run: rm -rf

Jak to použít pro tvůj projekt?

✓ Co doporučuji povolit

modify files in /src
modify files in /public
run: npm run dev
run: npm run build

✗ Co zakázat

delete any files
modify package.json
run: npm install
modify .env

⚠️
Pozor: Allow list je přísnější

Pokud máš Allow list — agent NIČEHO jiného nemůže dělat. Pokud máš Deny list — agent smí všechno kromě zakázaného. Validu pro receptář: Deny list.

// sekce 04

Chránění citlivých dat

Konkrétní věci které musíš chránit:

❌ NIKDY v projektu:

  • API klíče — Google, Firebase, atd.
  • Hesla — databáze, služby
  • Database URL — pokud je produkční
  • Privátní tokeny — JWT, OAuth tokeny

Jak je správně chránit?

✓ Bezpečný postup

☐ Vytvoř soubor .env.local — tam všechny klíče
☐ Přidej .env.local do .gitignore — nesmí jít na GitHub
☐ V Antigravity Settings → Project → Ignore Files → přidej .env.local
☐ Agent nebude vidět .env.local — ochrana

Pro receptář konkrétně:

.env.local tam patří:

# .env.local (NEVER na GitHub, nikdy pro agenta) VITE_GOOGLE_API_KEY=tvoj-tajny-klic NEXT_PUBLIC_SUPABASE_URL=https://... DATABASE_PASSWORD=super-tajne
💡
Env variables v kódu

V kódu pak užívaš process.env.VITE_GOOGLE_API_KEY — a ta se na runtime načte z .env.local. Agent vidí jen že tu je reference, ale ne hodnotu.

// sekce 05

Data privacy — kde běží kód a data

Kde se procesují tvé datos?

KomponentaKde běžíPrivátnost
Antigravity IDE Tvůj lokální počítač ✓ Lokální, nic se neodesílá
Tvůj kód (src/) Tvůj počítač + Gemini API ⚠️ Agent to vidí a může to poslat Google
Terminal výstup Tvůj počítač ✓ Lokální, pokud agent to nesdílí
Chrome Extension screenshoty Tvůj počítač + Gemini API ⚠️ Screenshot se může poslat Google (pokud ho agent sdílí)
Aplikace (localhost:3000) Tvůj počítač (ve VS Code) ✓ Lokální, není na internetu
🔴
Kód = data na Google serverech

Když agent pracuje, tvůj kód jde na Google API. To je normální pro Gemini/AI. Pokud máš velmi citlivý kód (na produkční data, bezpečnostní logiku) — měl bys si to uvědomit.

Local-only režim (pokud je dostupný)

Některé Antigravity verze mají "Local Only Mode" — agent pracuje bez posílání do cloudu. Pokud je to pro tebe kritické, zjisti si jestli je dostupné.

// sekce 06

Audit trail — coje se dělo

Antigravity si pamatuje co agent dělal. Můžeš si to prohlédnout.

Kde je historii akcí?

View → Agent History (nebo Command Palette → Show Agent Activity)

Co vidíš v historii?

  • Kdy se spustilo
  • Co agent dělal (plán)
  • Co přesně změnil (diff)
  • Co se stalo (build status, výstupy)
  • Byl-li schválen či ne
✓
Praktické užití: "Kontrola zpět"

Pokud si myslíš že agent něco podivného dělal — otevři historii, podívej se přesně co se změnilo. Pak můžeš git checkout nebo undo.

// sekce 07

Role-based access — pro týmy

Pokud by na projektu pracovalo více lidí (nyní pro tebe neaktuální, ale užitečné vědět):

Role které existují

  • Owner — tvůr. Všechna práva.
  • Editor — může měnit kód, spouštět agenty.
  • Reviewer — vidí změny, schvaluje. Nemůže sám měnit.
  • Viewer — jen čte, nic nezmění.

Pro tvůj projekt sám: Ty jsi Owner. Hotovo.

// sekce 08

Praktické bezpečnostní tipy

🔒 Tvůj checklist bezpečnosti

☐ Spusť Antigravity s Review-Driven režimem (ne Full Autonomy)
☐ Vytvoř .gitignore s .env.local a jeho zákaz v Antigravity
☐ Postav Deny list: delete, package.json, .env, npm install
☐ Před prací s agentem: git commit (záloha)
☐ Čti každý plán. Nepřeskakuj schvalování.
☐ Když agentneví nevím — řekni mu "Show me what you changed"
☐ Pravidelně kontroluj Agent History
☐ Nesdílej API klíče. Nikdy. Vůbec.
⚠️
Při prvního spuštění s agentem

Dej agentovi malý, nekritický task. Třeba "Přidej komentář do souboru README". Uvidíš jak pracuje, jak vypadá workflow. Pak když budeš jistý — komplexnější věci.

Když se agent choval divně:

  1. Podívej se do Agent History — co přesně dělal?
  2. git checkout . — vrátí vše do posledního commitu
  3. Znovu si sedni s agentem — co se stalo? Řekni mu pravidla znova.
