// sekce 01

Co je Manager View

Manager View je druhý mód Antigravity. Zatímco Editor View je "ty + jeden agent v panelu", Manager View je Mission Control pro orchestraci více agentů najednou.

Analogie: Orchestra

Editor View: Ty a dirigent (agent) hrají společně, ty máš poslední slovo.
Manager View: Ty jsi dirigent, máš více hudebníků (agentů) a všichni hrají podle tvého plánu zároveň.

Fyzicky co se změní?

  • Mizí bočný agent panel — místo toho máš big-screen dashboard
  • Vidíš všechny agenty najednou, jejich stavy, progress
  • Agenti pracují paralelně — jeden dělá feature A, druhý feature B, třetí testy — všichni zároveň
  • Ty jen sleduješ a řídíš — je to autopilot, ale ty jsi pilot
ℹ️
Pro tvůj receptář (zatím) není nutný

Hobby projekt, jeden člověk = Editor View stačí. Manager View se hodí když máš komplexní projekt kde by agenti mohli dělat věci nezávisle na sobě.

// sekce 02

Editor View vs Manager View — přesný rozdíl

Editor View (toto máš teď)
📝Jeden agent v panelu
🔄Sériově — jeden úkol za druhým
👁️Ty vidíš agent panel + editor vedle sebe
⚠️Když agent čeká na schválení, nic se nedělá
💡Jednodušší, méně složitý
Manager View (pro experty)
🤖Více agentů najednou
⚡Paralelně — všichni pracují zároveň
🎛️Vidíš velký dashboard se všema agenty
🚀Agent A pracuje, Agent B čeká, Agent C se chystá
🔧Pokročilejší, více kontroly, více komplexity

Praktický příklad

Editor View flow:

Ty: "Přidej login komponentu"
Agent dělá ✓

Ty: "Teď přidej database backup"
Agent dělá ✓

Manager View flow:

Ty: [defineš workflow:]
- Agent1: Přidej login komponentu
- Agent2: Přidej database backup (až bude Agent1 hotov)
- Agent3: Spusť testy (až bude Agent2 hotov)

Všichni agenti pracují optimálně a správné pořadí se řeší automaticky
// sekce 03

Spuštění prvního agenta v Manager View

Jak se do Manager View dostaneš?

Krok 1: Přepnout se do Manager View

V levém rohu Antigravity: View Selector → vyber "Manager View" (místo "Editor View")

Nebo: View → Manager Mode (menu)

Krok 2: Vidíš dashboard

Místo editoru se otevře Mission Control dashboard — velké obrazovka s:

  • Timeline agentů (vpravo)
  • Stav jednotlivých agentů (kdo pracuje, kdo čeká, kdo je hotov)
  • Workflow editor (uprostřed) — kde definuješ co se má stát

Krok 3: Vytvořit nový agent task

[+] Add Agent → pojmenuj agenta (třeba "LoginAgent") → co má dělat

Agent: LoginAgent Task: "Vytvoř login komponentu s email/password fieldem" Status: [⏳ Waiting for input]

Krok 4: Spustit workflow

[RUN] tlačítko uprostřed → všichni agenti se spustí

✓
Poprvé: Je to lehce chaotické. Dashboard má spoustu informací. Počkej chvíli, až si zvykneš.
// sekce 04

Paralelní agenti — jak se orchestrují

Klíčová myšlenka: Agent A nemusí čekat na Agenta B pokud jsou nezávislí.

Závislosti (Dependencies)

Někdy agent MUSÍ čekat:

Agent1 (Login komponenta) ↓ Agent2 (Database pro uživatele) ← čeká na Agent1 ↓ Agent3 (Testy) vs. Agent1 (Login) ↗ ↘ Agent2 (Styling) Agent3 (Tests) ← paralelně, nezávislí ↘ ↙ (All done)

Jak to definovat?

V Manager View workflow editoru:

agents: - name: LoginFeature task: "..." - name: DatabaseSetup task: "..." depends_on: [LoginFeature] ← čeká! - name: Styling task: "..." # no depends_on = může běžet hned paralelně

Výhoda

  • Když Agent1 čeká na schválení → Agent2 a Agent3 pracují
  • Výrazně se zrychluje vývoj
  • Komplexní projekty se řeší "jako by to dělali 3 lidi najednou"
⚠️
Pozor na konflikty souborů

Pokud Agent1 a Agent2 oba editují package.json → konflikt v Gitu. Musíš dbát aby paralelní agenti pracovali na různých souborech.

// sekce 05

Mission Control — dashboard a monitoring

Dashboard ci ukazuje všechno co se děje.

Levá strana — Agenti & States

  • 🟢 Agent1: Running — teď pracuje
  • 🟡 Agent2: Waiting — čeká na schválení (tvůj input)
  • ⚫ Agent3: Blocked — čeká na Agent1
  • ✅ Agent4: Complete — hotov

Uprostřed — Timeline

Grafická linie co se dělo v čase. Vidíš:

  • Kdy se každý agent spustil
  • Jak dlouho dělal co dělal
  • Kde byly čekací doby
  • Topologie (kdo čekal na koho)

Pravá strana — Artifacts & Logs

  • Klikni na agenta → vidíš jeho plan, diffs, error zprávy
  • Terminal output — co se dělalo
  • Build logs — výsledky

Interakce

  • [Approve] — schvál agenta, ať pokračuje
  • [Edit] — změň plán agenta
  • [Reject] — zastavit, agent se zeptá co jinak
  • [Pause All] — zastavit všechny agenty
  • [Restart] — znovu začít konkrétní agenta
// sekce 06

DAG — Workflow Definition (pro experty)

DAG = Directed Acyclic Graph. To je fancy slovo pro "definici kdo závisí na kom".

Jak se to píše?

V YAML nebo JSON:

workflow: name: "Receptář - Přidání Feature" agents: - id: agent-1 name: "Features" task: "Přidej filtr na cenu" timeout: 600s - id: agent-2 name: "Tests" task: "Napiš testy pro filtr" depends_on: [agent-1] - id: agent-3 name: "Docs" task: "Aktualizuj README" depends_on: [agent-2]

Výsledek

Antigravity si z toho vytvoří graf:

Agent1 → Agent2 → Agent3

Spustí Agent1, až bude hotov → Agent2, až bude hotov → Agent3.

ℹ️
Pro začátečníky: Můžeš ignorovat

DAG se vytváří automaticky když klikáš "[+] Add Agent" a vztahuješ je myší. YAML je jen pro pokročilé co chtějí úplnou kontrolu.

// sekce 07

Praktický příklad — komplexní workflow

Představ si že na receptáři potřebuješ udělat toto:

  • Přidat filtr na cenu (feature)
  • Stylem to zatím netřešíš, jen funkčnost

V Editor View:

Ty: Přidej filtr
Agent dělá → hotov → Ty: OK

Ty: Teď nastyluj
Agent dělá → hotov → Ty: OK

Ty: Napiš testy
Agent dělá → hotov

Čas: 30 minut (sériově)

V Manager View:

Ty vytvoříš 3 agenty:
Agent1: Přidej filtr
Agent2: Nastyluj (čeká na Agent1)
Agent3: Napiš testy (čeká na Agent2)

Spustíš [RUN]
- Agent1 pracuje
- Ty zatím [Approve] ostatní chystáš
- Agent1 hotov, Agent2 se spustí
- Agent2 pracuje, Agent3 se čeká
- Agent2 hotov, Agent3 se spustí

Čas: 25 minut (lepší paralelizace schválení)

Ještě pokročilejší:

Pokud by sytling a testy byly nezávislé (nemusí čekat na sebe):

Agent1: Filtr (2 min)
↓
Agent2: Styling (5 min)
Agent3: Testy (4 min) ← paralelně s Agent2

Čas: 11 minut! (místo 30)
// sekce 08

Kdy Manager View použít

✓ Dělej Manager View když:

  • Máš více nezávislých tasků — styling + testing + features
  • Chceš optimalizovat čas — paralelní práce je jednoznačně rychlejší
  • Máš komplexní projekt — více souborů, více komponent
  • Chceš vidět big picture — jak to všechno funguje dohromady

✗ Nepoužívej Manager View když:

  • Máš malý projekt (receptář atd.) — overhead
  • Tasks jsou vysoce závislé — jedna věc se musí dělat po druhé
  • Jsi nový v Antigravity — nejdříve nauč se Editor View
  • Pracuješ sám — není důvod pro orchestraci

Pro tvůj projekt (receptář)

Editor View stačí. Projekt je dostatečně malý, tasks jsou spíš sekvenční (přidej feature → nastyluj → test). Manager View je pro pokročilejší:

  • Velké projekty s více formování paralelními streamy
  • Týmy kde více agentů pracuje na různých partech
  • Optimalizace: frontend + backend + testing paralelně
ℹ️
V budoucnu — pokud se rozšíříš

Pokud se receptář stane komplexnějším projektem nebo budeš mít víc věcí co dělat — přepni na Manager View. Zatím je to overkill.
