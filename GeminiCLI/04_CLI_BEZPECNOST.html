Co přesně instaluješ

Je fér přesně vědět co si dáváš na počítač. Jsou to dva oddělené nástroje:

Node.js

Node.js je open-source runtime pro JavaScript. Vyvinul ho Ryan Dahl v roce 2009, dnes ho spravuje nadace OpenJS Foundation. Používají ho miliony vývojářů po celém světě – je to de facto standard pro vývoj webových aplikací. Tvůj projekt (rodinny-receptar) ho už používá – bez Node.js by ses s projektem vůbec nedostal dál.

  • Zdroj: nodejs.org – oficiální stránka, ne žádná podezřelá stránka třetí strany
  • Kód: Plně open-source, kdokoliv ho může zkontrolovat na GitHubu
  • Co dělá: Umožňuje spouštět JavaScript mimo prohlížeč – na tvém počítači
  • Co nedělá: Samo o sobě nekomunikuje s internetem, nic neodesílá

Gemini CLI

Gemini CLI je nástroj od Google LLC, vydaný pod licencí Apache 2.0 (otevřená licence). Instaluje se přes npm (správce balíčků Node.js) z oficiálního registru npmjs.com.

  • Zdroj: github.com/google-gemini/gemini-cli – oficiální Google repozitář
  • Kód: Open-source, veřejně dostupný, pravidelně auditovaný komunitou
  • Co dělá: Posílá tvoje dotazy na Google Gemini API a vrací odpovědi
  • Co nedělá: Nespouští se na pozadí, nepracuje bez tvého vědomí, neinstaluje žádné systémové služby
💡
Není to program který "běží na pozadí"

Gemini CLI se spustí pouze když ty napíšeš gemini do terminálu. Když terminál zavřeš, skončí. Není to žádný daemon, služba ani autostart. Na procesech počítače se neobjeví pokud ho sám nespustíš.

npm balíčky – co se instaluje s Gemini CLI

Při instalaci příkazem npm install -g @google/gemini-cli se nainstaluje nejen Gemini CLI, ale i jeho závislosti – knihovny které potřebuje k fungování. Typicky to bývá stovka nebo více balíčků.

⚠️
Stovky balíčků – proč a je to problém?

Každý moderní npm balíček závisí na dalších. Je to běžná praxe celého ekosystému Node.js. Není to nic neobvyklého ani nebezpečného – tvůj vlastní projekt (rodinny-receptar) má stovky závislostí v node_modules. Ty balíčky jsou standardní nástroje jako HTTP klient, parser JSON, šifrovací knihovny atd.

Oprávnění PowerShellu – co jsme změnili a proč

Toto je část, kde máš právo být opatrný. Pojďme si to rozebrat do detailu.

Co je Execution Policy a proč existuje

Windows má zabudovanou ochranu, která zabraňuje spouštění PowerShell skriptů (.ps1 souborů). Tato ochrana existuje proto, aby malware nemohl snadno spustit škodlivý skript. Ve výchozím stavu je nastavena na Restricted – žádné skripty nesmí běžet.

Problém: npm (správce balíčků) používá .ps1 skripty pro svoji práci. Bez povolení těchto skriptů npm prostě nefunguje.

Jaké jsou možnosti nastavení

🔴 Restricted (výchozí)

Žádné skripty nesmí běžet. Nejbezpečnější, ale nefunguje npm.

🟢 RemoteSigned (co jsme nastavili)

Lokální skripty (z tvého počítače) smí běžet. Skripty stažené z internetu musí mít digitální podpis od důvěryhodného vydavatele.

🟡 AllSigned

Každý skript musí mít digitální podpis. Bezpečnější než RemoteSigned, ale komplikovanější v praxi.

🔴 Unrestricted

Vše smí běžet bez omezení. Nebezpečné – nikdy toto nenastavuj.

Co přesně jsme změnili – scope CurrentUser

Klíčové je, že jsme změnili politiku pouze pro scope CurrentUser – tedy jen pro tvůj uživatelský účet, ne pro celý systém. Co to znamená v praxi:

✅
Co jsme NEUDĚLALI (důležité)
  • Nezměnili jsme nastavení pro LocalMachine (celý systém) – to zůstalo beze změny
  • Nezměnili jsme nastavení pro ostatní uživatele počítače
  • Nenastavili jsme Unrestricted – to by bylo nebezpečné
  • Neotevřeli jsme žádný port, nevytvořili žádné sdílení
⚠️
Co RemoteSigned DOVOLUJE (o čem musíš vědět)
  • Lokální .ps1 skripty na tvém počítači se mohou spouštět
  • To znamená: pokud stáhneš a spustíš podezřelý .ps1 soubor, Windows tě nezastaví
  • Stažené skripty bez digitálního podpisu stále nejdou spustit (to je ochrana)

Jak ověřit aktuální stav oprávnění

# Zobraz kompletní přehled oprávnění pro všechny scope: Get-ExecutionPolicy -List # Výsledek by měl vypadat takto (po naší změně): Scope ExecutionPolicy ----- --------------- MachinePolicy Undefined UserPolicy Undefined Process Undefined CurrentUser RemoteSigned ← jen toto jsme změnili LocalMachine Undefined ← systém zůstal nezměněn

Co Gemini CLI vidí a co posílá Googlu

Toto je klíčová otázka. Budu maximálně přímý.

Co Gemini CLI MŮŽE přečíst

Gemini CLI pracuje ve složce kde ho spustíš. Má přístup ke:

  • Všem souborům ve složce projektu – to je záměrné, proto je tak užitečný
  • Souborům ve složkách výše – pokud o ně požádáš nebo se jich sám zeptá
  • Obsahu schránky – pokud mu sám vložíš text
  • Výstupu příkazů – když spustí npm nebo jiný příkaz, vidí výsledek
🔴
Nikdy nedávej Gemini CLI tyto věci
  • Hesla, API klíče, tokeny – ani v souborech které "jen ukázat"
  • Přihlašovací údaje k bankám, emailu, čemukoliv citlivému
  • Soubor .env pokud obsahuje produkční klíče (Gemini API key, databáze)
  • Soubory z jiných složek kde máš citlivá data

Co se posílá Googlu

Gemini CLI je klient pro Google Gemini API. Komunikace probíhá takto:

  1. Ty napíšeš dotaz (prompt) do terminálu
  2. Gemini CLI přidá kontext (co přečetl ze souborů) a odešle vše na api.googleapis.com
  3. Google Gemini API zpracuje dotaz a vrátí odpověď
  4. Odpověď se zobrazí v tvém terminálu
⚠️
Obsah souborů které Gemini čte jde přes Google servery

Kód tvého receptáře, struktura projektu, texty – vše co Gemini přečte a zahrne do kontextu se odesílá na Google. Pro hobby projekt jako receptář je to naprosto v pořádku. Pro komerční projekt s citlivými daty by to byl problém.

✅
Jak Google data zpracovává (podle jejich podmínek)
  • Data jsou šifrována při přenosu (HTTPS/TLS)
  • Google tvrdí že data z bezplatného API tier používá ke zlepšení modelů
  • Plné podmínky: ai.google.dev/gemini-api/terms

Co Gemini CLI NEodesílá

  • Soubory mimo složku projektu (pokud mu to sám neřekneš)
  • Systémové informace o počítači (verze Windows, nainstalované programy...)
  • Hesla uložená v prohlížeči nebo správci hesel
  • Cokoliv co mu explicitně neřekneš nebo co sám nespustí

Celkové hodnocení pro tvůj případ použití

Používáš Gemini CLI pro hobby projekt (receptář) bez citlivých dat, bez hesel v kódu, bez produkčních databází. V tomto kontextu je bezpečnostní riziko srovnatelné s instalací jakéhokoliv softwaru z důvěryhodného zdroje – jako když instaluješ VLC nebo LibreOffice.

Zlatá pravidla bezpečnosti při práci s Gemini CLI

✅ Vždy dělej toto

☐Pracuj vždy pouze ve složce projektu – nespouštěj Gemini CLI v kořeni disku C: nebo v systémových složkách
☐Čti každé potvrzení které Gemini vyžaduje před spuštěním příkazu nebo změnou souboru – nikdy neklikej "ano na vše" naslepo
☐Udělej git commit (zálohu) před každou prací – takže ztráta dat při chybě = max. hodina práce
☐Soubor .env (pokud ho projekt má) přidej do .gitignore – nesmí se dostat ani do Gitu ani ke Gemini CLI pokud obsahuje produkční klíče
☐Po skončení práce Gemini CLI vždy ukonči příkazem /quit – nenechávej ho zbytečně běžet

🚫 Nikdy nedělej toto

✗Neukládej hesla, API klíče ani tokeny přímo do souborů projektu – pokud tam jsou, Gemini CLI je bude znát a odešle Googlu
✗Nespouštěj Gemini CLI jako správce (Administrator) – pro běžnou práci to není potřeba a zvyšuje to rizika
✗Nenastavuj PowerShell execution policy na Unrestricted – RemoteSigned která máš nastavena je správná volba
✗Neinstaluj Gemini CLI z neoficiálních zdrojů – pouze npm install -g @google/gemini-cli z officiálního npm registru
✗Nesouhlasíš-li s tím co Gemini navrhuje – odmítni. AI dělá chyby. Ty jsi ten kdo rozhoduje.

Jak odinstalovat Gemini CLI

Odinstalace je jednoduchá, čistá a úplná. Jeden příkaz vše odstraní.

1
Odinstaluj Gemini CLI přes npm
npm uninstall -g @google/gemini-cli

Příznak -g říká npm aby odstranil globálně instalovaný balíček. Uvidíš výstup "removed X packages".

2
Ověř že byl odstraněn
gemini --version # Měl bys dostat: "The term 'gemini' is not recognized" # nebo podobnou chybu – to znamená že byl úspěšně odstraněn
3
Odstraň konfiguraci a historii (volitelné)

Gemini CLI si ukládá konfiguraci a historii konverzací do skryté složky ve tvé domovské složce. Pro úplné čistý odinstalaci:

# Windows – zobraz cestu k home složce: echo $HOME # Typicky: C:\Users\marti # Smaž skrytou složku .gemini (v File Exploreru je skrytá): Remove-Item -Recurse -Force "$HOME\.gemini" # Pokud složka neexistuje, dostaneš chybu – to je v pořádku
✅
Co se odinstalací ODSTRANÍ
  • Všechny soubory Gemini CLI a jeho závislostí z npm global složky
  • Příkaz gemini přestane být dostupný v terminálu
  • Složka .gemini s historií (pokud ji odstraníš ručně)
💡
Co se odinstalací NEodstraní
  • Node.js – zůstane (odstraníš ho zvlášť v sekci 07)
  • npm – zůstane (je součástí Node.js)
  • Soubory tvého projektu – těch se odinstalace vůbec nedotkne
  • Přihlášení k Google účtu – zruší se automaticky se smazáním složky .gemini

Jak odinstalovat Node.js

⚠️
Pozor: Tvůj projekt Node.js potřebuje!

Projekt rodinny-receptar ke svému vývoji Node.js vyžaduje (npm run dev, build, atd.). Odinstaluj Node.js pouze pokud chceš přestat s projektem úplně, nebo pokud ho máš jinak zajištěný.

Windows – odinstalace přes Nastavení

1
Otevři Nastavení → Aplikace

Start → Nastavení → Aplikace → Nainstalované aplikace (nebo "Přidat nebo odebrat programy" ve starším Windows)

2
Najdi Node.js a odinstaluj

Hledej "Node.js" v seznamu. Klikni na tři tečky (nebo Odinstalovat) → Odinstalovat. Průvodce odinstalací odstraní Node.js i npm.

3
Volitelné – odstraň zbytky

Instalátor občas nechá pozůstatky. Pro úplnou čistotu v PowerShellu (jako správce):

# Odstraň Node.js složky pokud existují: Remove-Item -Recurse -Force "C:\Program Files\nodejs" -ErrorAction SilentlyContinue Remove-Item -Recurse -Force "$HOME\AppData\Roaming\npm" -ErrorAction SilentlyContinue Remove-Item -Recurse -Force "$HOME\AppData\Roaming\npm-cache" -ErrorAction SilentlyContinue
4
Ověř odinstalaci
node --version # Měl bys dostat: "The term 'node' is not recognized" → úspěch

Linux – odinstalace

# Ubuntu / Debian: sudo apt-get remove --purge nodejs npm sudo apt-get autoremove # Pokud jsi použil nvm: nvm uninstall node # Pak odstraň nvm samotné: rm -rf "$HOME/.nvm" # A odstraň řádky o nvm z ~/.bashrc nebo ~/.zshrc
// sekce 08

Jak vrátit oprávnění PowerShellu do původního stavu

Pokud chceš vrátit execution policy pro svého uživatele zpět do výchozího stavu, je to jeden příkaz. Spusť PowerShell normálně (ne jako správce – tohle nastavení je pro tvého uživatele):

# Vrátí CurrentUser policy do původního stavu (Undefined = výchozí): Set-ExecutionPolicy -ExecutionPolicy Undefined -Scope CurrentUser # Potvrď stisknutím Y # Ověř: Get-ExecutionPolicy -Scope CurrentUser # Měl bys vidět: Undefined
✅
Po tomto příkazu je vše jako před instalací

Execution policy pro tvého uživatele je vrácena na Undefined – systém pak použije nastavení LocalMachine (které jsme nikdy nezměnili). Efektivně jsi zpět ve výchozím stavu.

// sekce 09

Kompletní čistá odinstalace – vše najednou

Pokud chceš smazat vše a vrátit počítač do stavu před instalací, proveď tyto kroky v tomto pořadí:

1
Ukonči Gemini CLI pokud běží
/quit
2
Odinstaluj Gemini CLI přes npm
npm uninstall -g @google/gemini-cli
3
Smaž konfiguraci Gemini CLI
Remove-Item -Recurse -Force "$HOME\.gemini" -ErrorAction SilentlyContinue
4
Vrať PowerShell execution policy
Set-ExecutionPolicy -ExecutionPolicy Undefined -Scope CurrentUser

Potvrď Y.

5
Odinstaluj Node.js (přes Nastavení Windows)

Start → Nastavení → Aplikace → hledej "Node.js" → Odinstalovat

6
Odstraň zbytky npm (volitelné)
Remove-Item -Recurse -Force "$HOME\AppData\Roaming\npm" -ErrorAction SilentlyContinue Remove-Item -Recurse -Force "$HOME\AppData\Roaming\npm-cache" -ErrorAction SilentlyContinue
7
Ověř že je vše pryč
node --version ← měl bys dostat "not recognized" gemini --version ← měl bys dostat "not recognized" Get-ExecutionPolicy -Scope CurrentUser ← měl bys dostat "Undefined"
✅
Po těchto 7 krocích je počítač ve stejném stavu jako před instalací.

Žádné zbytky, žádné pozadní procesy, žádné změněné systémové nastavení. Soubory projektu rodinny-receptar zůstanou nedotčeny – odinstalace se jich nedotkne.

Výsledný verdikt: je to bezpečné?

Krátká odpověď: Pro tvůj případ použití – ANO.

✅ Co svědčí PRO bezpečnosti
✅Gemini CLI neběží na pozadí – spouštíš ho sám, sám ho zavíráš
✅Open-source kód od Googlu, veřejně auditovatelný komunitou
✅Nevytváří žádné systémové služby, porty ani sdílení
✅Ptá se tě před každou změnou souboru nebo spuštěním příkazu

⚠️Kód projektu jde přes Google – ale pro receptář to nevadí

Srovnání s čím běžně pracuješ

Google AI Studio také posílá tvůj kód Googlu – a to jsi používal předtím. VS Code, Chrome, Windows Update – všechny tyto nástroje komunikují s externími servery. Gemini CLI v tomto ohledu není výjimkou, pouze o tom mluví otevřeně.
